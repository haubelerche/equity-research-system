# CALCULATION_MODEL_AUDIT_PLAN_FOR_CLAUDE.md

> **Scope:** Vietnam Pharma Equity Research multi-agent system.  
> **Purpose:** Audit the current calculation model before any remediation or refactor.  
> **Primary question:** How does the current system calculate every number shown in the financial report, and can each number be traced back to source evidence, deterministic formula code, and final report output?  
> **Non-goal:** Do not fix code in this phase. Do not optimize layout. Do not rewrite report narrative. Do not hard-code ticker-specific exceptions.

---

## 0. Why this audit is required

The current master remediation plan is still useful as the target architecture, but it is not enough to start implementation immediately. Before fixing the system, we must know exactly how the current pipeline calculates financial statement metrics, ratios, forecast lines, FCFF, FCFE, WACC/Re, target price, upside/downside, and rating.

The highest-risk failure is not only that the report looks weak. The real risk is that the system may generate report-looking outputs while the underlying calculation model is opaque, inconsistent, partly inferred by fallback logic, or accidentally generated by LLM prose.

This audit must answer:

1. Which numbers are sourced from filings, APIs, secondary sources, or manual/fallback data?
2. Which numbers are calculated by deterministic Python code?
3. Which numbers are inferred, defaulted, hard-coded, or written by LLM?
4. Which formulas are currently implemented, where, and with what assumptions?
5. Whether FCFF and FCFE can be recomputed from structured artifacts.
6. Whether final report numbers can be traced back to exact source artifacts and formula traces.

No remediation phase should begin until this audit is completed.

## 3. Audit standard: classification of every number

Every important number must be classified into exactly one of these categories:

| Classification | Meaning |
|---|---|
| `OFFICIAL_SOURCE` | Directly extracted from official filing/disclosure with source metadata. |
| `SECONDARY_SOURCE` | Taken from a reputable secondary source. |
| `CONVENIENCE_API` | Taken from API/vendor data without official reconciliation. |
| `DETERMINISTIC_FORMULA` | Calculated by code from traceable inputs. |
| `FORECAST_ASSUMPTION` | Produced by explicit forecast assumption or scenario method. |
| `FALLBACK_ESTIMATE` | Filled by fallback heuristic due to missing data. |
| `HARDCODED_DEFAULT` | Hard-coded constant or static default. |
| `LLM_GENERATED` | Created or modified by LLM prose/prompt. |
| `UNTRACED` | Cannot be traced to source, formula, artifact, or code path. |

Trust rule:

```text
Only OFFICIAL_SOURCE, SECONDARY_SOURCE with reconciliation, DETERMINISTIC_FORMULA, and explicit FORECAST_ASSUMPTION can be considered potentially trusted.

CONVENIENCE_API-only, FALLBACK_ESTIMATE, HARDCODED_DEFAULT, LLM_GENERATED, and UNTRACED values are not trusted for final valuation unless explicitly approved and documented.
```

## 8. Phase E — FCFF and FCFE audit

### Objective

Verify whether FCFF and FCFE are separately and correctly calculated.

### Expected FCFF logic

```text
FCFF = EBIT × (1 - tax_rate)
     + depreciation
     - capex_positive
     - delta_nwc
```

### Expected FCFE logic

```text
FCFE = net_income
     + depreciation
     - capex_positive
     - delta_nwc
     + net_borrowing
```

### Required checks

Claude must verify:

```text
- FCFF uses WACC.
- FCFE uses Re/cost of equity.
- FCFF does not use equity cash flow.
- FCFE does not use WACC.
- CAPEX sign convention is consistent.
- delta_nwc sign convention is consistent.
- net_borrowing comes from a debt schedule.
- terminal value is computed separately for FCFF and FCFE.
- WACC <= terminal_growth blocks valuation.
- Re <= terminal_growth blocks valuation.
- valuation does not auto-cap terminal growth silently.
```

### Required output

File:

```text
audits/-1_fcff_fcfe_audit.md
```

Minimum table:

| Component | FCFF current behavior | FCFE current behavior | Expected behavior | Code path | Risk | Required fix |
|---|---|---|---|---|---|---|

Special focus: net borrowing

Claude must explicitly audit:

```text
new_borrowing
debt_repayment
beginning_debt
ending_debt
short_term_debt
long_term_debt
lease liabilities if present
whether net_borrowing is derived from CFS or balance sheet delta
whether signs are consistent
whether missing values are defaulted
```

Common FCFE failure patterns to detect:

```text
- net_borrowing sign reversed.
- debt repayment treated as total ending debt.
- CAPEX subtracted twice.
- signed negative CAPEX is subtracted again.
- delta_nwc sign reversed.
- interest and debt schedule are disconnected.
- shares are wrong by 1,000x or 1,000,000x.
```
